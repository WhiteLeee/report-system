# Report System 多角色治理分析设计方案

## 1. 文档目的

本文档用于明确 `report-system` 数据分析模块在“门店、加盟商、运营组织”三个核心业务主体下的关系模型、责任模型和行动驱动模型。

本文档重点回答以下问题：

1. 当前数据分析设计是否完整表达了门店、加盟商、运营组织三者关系
2. 当前分析结果是否足够有价值
3. 当前分析结果是否能驱动各角色执行改善动作
4. 后续分析模块应如何补齐关系维度、责任维度和行动型指标

## 2. 当前判断结论

### 2.1 当前设计是否有价值

有价值，但当前价值主要体现在：

- 能看巡检规模
- 能看巡检质量分布
- 能看问题类型和组织 / 门店排行
- 能看人工复核效率
- 能看整改单闭环情况

也就是说，当前设计已经具备“管理统计首页”的基础能力。

### 2.2 当前设计的主要不足

当前设计还不能视为完整的“治理分析体系”，原因如下：

1. `加盟商` 已成为正式分析维度，但治理动作表达仍不够完整
2. `运营组织` 当前更多是展示维度，不是责任归属维度
3. `门店 / 加盟商 / 运营组织` 三者关系在 analytics facts 中尚未完整建模
4. 当前指标以统计型指标为主，缺少能驱动改善动作的行动型指标

### 2.3 当前设计是否能驱动各角色行动

- `门店`
  - 可以部分驱动
  - 因为结果详情、复核、整改单闭环已经存在
  - 但 analytics 首页对门店仍缺“待办视图”

- `加盟商`
  - 当前可以部分驱动
  - 因为已经有正式的加盟商聚合、加盟商排行、加盟商闭环率与高风险加盟商识别
  - 但仍缺超期加盟商、加盟商待办和更完整的 drill-down 治理链

- `运营组织`
  - 可以部分驱动
  - 因为已有组织排行、门店排行、整改超期排行
  - 但仍缺明确的督导优先级、责任对象和持续恶化识别

结论：

`当前 analytics 设计可以支撑“统计”和“发现异常”，但还不能完整支撑“责任归属”和“治理动作”。`

## 3. 三类主体的关系模型

### 3.1 业务主体

当前分析必须围绕三类主体建立关系：

- `门店`
  - 巡检发生地
  - 问题发生地
  - 整改执行地

- `加盟商`
  - 经营责任主体
  - 对其名下门店经营质量、执行质量、整改质量负责

- `运营组织`
  - 管理 / 督导 / 支持主体
  - 对其辖区内加盟商和门店的治理结果负责

### 3.2 关系链建议

建议将数据关系明确表达为：

```text
企业
  -> 运营组织
    -> 加盟商
      -> 门店
        -> 巡检结果 / 问题 / 复核 / 整改单
```

### 3.3 当前数据现状

当前主数据已经具备：

- `store_id`
- `store_name`
- `organize_code`
- `organize_name`
- `franchisee_name`

但 analytics 层当前稳定保留的主要只有：

- `store_id`
- `store_name`
- `organization_code`
- `organization_name`
- `franchisee_name`

因此当前 analytics 已进入：

`门店、加盟商、运营组织三方都可聚合，但责任归属与治理动作表达仍需继续增强`

## 4. 统计维度与责任维度

### 4.1 统计维度

统计维度用于回答“问题发生在哪里、分布如何”。

当前建议包括：

- 时间
- 企业
- 运营组织
- 加盟商
- 门店
- 报告类型
- 报告主题
- 巡检计划
- 问题类型
- 巡检结论
- 复核状态
- 整改状态

### 4.2 责任维度

责任维度用于回答“谁应该改、谁应该跟、谁应该督导”。

当前建议包括：

- `门店`
  - 对问题处理和整改执行负责

- `加盟商`
  - 对名下门店整体质量和整改质量负责

- `运营组织`
  - 对辖区内门店和加盟商的治理推进、督导与闭环效率负责

### 4.3 为什么要拆开

如果不拆开，分析页只能告诉用户：

- 哪些门店问题多
- 哪些组织问题多

但不能告诉用户：

- 哪些加盟商要重点跟
- 哪些运营组织在“督而不改”
- 哪些门店问题虽然发生在一处，但责任应追到加盟商

因此后续 analytics 模型必须同时支持：

- `发生视角`
- `责任视角`

## 5. 三类主体的核心分析问题

### 5.1 门店视角

门店需要回答：

1. 我当前有哪些问题待处理
2. 哪些问题已经复核
3. 哪些整改单还没完成
4. 我是否连续多次出现问题
5. 我最近的巡检质量是否在变差

### 5.2 加盟商视角

加盟商需要回答：

1. 我名下有多少门店在出问题
2. 哪些门店问题最多
3. 哪类问题最集中
4. 我的整改闭环率如何
5. 我的超期门店有多少
6. 我与其他加盟商相比处于什么水平

### 5.3 运营组织视角

运营组织需要回答：

1. 哪些辖区风险最高
2. 哪些加盟商长期表现较差
3. 哪些门店反复异常
4. 复核积压卡在哪
5. 整改闭环慢在哪里
6. 哪些对象需要督导和升级跟进

## 6. 统计型指标与行动型指标

### 6.1 统计型指标

统计型指标用于回答：

- 发生了多少
- 分布如何
- 趋势如何

例如：

- 问题总数
- 巡检失败数
- 无法判定占比
- 问题类型 Top N

### 6.2 行动型指标

行动型指标用于回答：

- 现在谁需要立刻处理
- 应该优先抓哪里
- 哪些对象已经持续恶化

后续必须补的行动型指标建议包括：

- 待人工复核门店数
- 超期未整改门店数
- 连续异常门店数
- 重复问题门店数
- 高风险加盟商数
- 加盟商闭环率排行
- 督导积压组织数
- 组织辖区内超期加盟商数

## 7. 对 analytics facts 的补齐要求

### 7.1 当前必须补齐的字段

后续 analytics facts 至少要稳定补齐：

- `organization_code`
- `organization_name`
- `franchisee_name`

如果主数据后续可提供，建议进一步补齐：

- `franchisee_id`

### 7.2 组织维度必须使用稳定编码

当前 analytics 中大量使用 `organization_name` 聚合。

这会带来风险：

- 组织名称变更
- 组织名称重复
- 企业间重名

因此后续组织维度聚合必须优先使用：

- `organization_code`

而 `organization_name` 仅作为展示字段。

## 8. 对分析页面的改造要求

### 8.1 页面不应只有单一统计视角

后续页面必须明确支持：

- 门店治理视角
- 加盟商治理视角
- 运营组织治理视角

### 8.2 页面应从统计首页升级为治理首页

后续页面的核心问题应变成：

1. 哪些门店待处理
2. 哪些加盟商风险最高
3. 哪些运营组织需要重点督导
4. 哪些问题在重复发生
5. 哪些整改迟迟不闭环

### 8.3 建议的页面结构调整方向

在现有“经营概览、巡检质量、问题分析、复核与整改、组织与门店定位”结构上，后续应进一步强化：

- `加盟商治理区`
- `治理优先级区`
- `持续恶化区`

---

## 9. 建议的下一轮建设重点

### 9.1 第一层：补齐关系维度

先做数据底座补齐：

1. 在 analytics facts 中补齐 `organization_code`
2. 在 analytics facts 中补齐 `franchisee_name`
3. 在 analytics filters 中新增 `franchisee`
4. 新增加盟商聚合查询
5. 新增加盟商排行和加盟商闭环分析

### 9.2 第二层：补齐行动导向

在页面和指标上补齐治理动作：

1. 新增高风险加盟商识别
2. 新增超期门店 / 超期加盟商识别
3. 新增重复问题门店识别
4. 新增组织督导积压识别
5. 将排行从“统计排行”升级为“治理优先级列表”

---

## 10. 结论

当前 `report-system` 的数据分析设计：

- 作为“统计分析首页”已经成立
- 作为“巡检数据分析与治理平台”还不完整

当前最大缺口不是图表数量，而是：

1. `加盟商` 没有成为正式分析主体
2. `组织` 还没有稳定编码化
3. 缺少责任归属视角
4. 缺少行动型指标

因此后续分析模块应明确进入下一阶段：

`从统计分析，升级为多角色治理分析。`
