# 慧运营整改单自动下发技术方案

## 1. 设计目标

自动下发能力需要满足两个要求：

1. 与现有人工复核提交流程解耦。
2. 后续增加新规则时，只改规则层，不改主流程。

因此本次采用“三层结构”：

1. 配置层
2. 规则层
3. 执行层

## 2. 结构分层

### 2.1 配置层

文件：

- `backend/system-settings/system-settings.types.ts`
- `backend/system-settings/system-settings.service.ts`
- `backend/system-settings/sqlite-system-settings.repository.ts`
- `app/admin/settings/page.tsx`
- `app/admin/settings/save/route.ts`

职责：

1. 保存自动下发开关和策略参数。
2. 提供统一读取接口。
3. 不和具体规则实现耦合。

### 2.2 规则层

文件：

- `backend/rectification/auto-rectification.rules.ts`
- `backend/rectification/auto-rectification.types.ts`

职责：

1. 定义自动下发候选上下文。
2. 定义规则接口 `AutoRectificationRule`。
3. 根据配置选择规则并输出命中结果。

当前规则：

- `issue_found`

### 2.3 执行层

文件：

- `backend/rectification/auto-rectification.service.ts`

职责：

1. 读取系统设置。
2. 获取报告明细。
3. 对每条巡检结果生成候选上下文。
4. 调用规则层判断是否应下发。
5. 调用已有 `RectificationService.createOrdersForReview()` 创建整改单。
6. 汇总执行结果并返回摘要。

## 3. 触发链路

当前触发入口：

- `app/api/reports/publish/route.ts`

流程：

1. `reportService.publishReport(parsed.data)`
2. 报告入库成功后
3. `autoRectificationDispatchService.processPublishedReport(reportId)`
4. 自动下发结果写入发布响应 `auto_rectification`

重要原则：

1. 自动下发异常不会中断报告发布成功响应。
2. 发布接口仍返回 201/200。
3. 自动下发摘要附加到响应体中。

## 4. 自动下发候选构造

每个候选基于单条 `ReportResult` 构建：

1. `result`
2. `issues`
3. `inspections`
4. `report`

结果关联逻辑：

1. 优先按 `result_id` 关联。
2. 如果没有 `result_id`，按 `metadata.capture_id` 回退匹配。

这样可以与当前人工复核接口保持一致的结果聚合口径。

## 5. 当前规则实现

### 5.1 issue_found 规则

规则代码：

- `issue_found`

命中条件：

1. `classifyReportResultSemantics(...) === "issue_found"`
2. 结果关联问题项数量大于 0

命中后执行：

1. 取当前结果关联问题项作为 `selectedIssues`
2. 取当前结果图片作为 `imageUrls`
3. `description = 问题项列表 + 自动备注`
4. 调用 `RectificationService.createOrdersForReview(...)`

## 6. 去重策略

当前去重策略：

1. 以 `result_id` 为粒度检查已有整改单。
2. 若系统设置 `autoRectificationSkipIfExists = true`，则跳过当前结果。

这样可避免报告重复发布或重复处理时反复建单。

## 7. 与人工复核链路的关系

人工链路仍然保留：

- `app/api/reports/[reportId]/images/[imageId]/review-status/route.ts`

自动链路不直接调用该接口，也不依赖前端页面状态。

两者关系：

1. 人工复核链路
   - 面向人工确认
   - 支持选问题项、备注、确认弹窗

2. 自动下发链路
   - 面向发布后自动执行
   - 使用系统配置和规则自动生成建单内容

这样后续任一链路调整都不会直接拖动另一条链路。

## 8. 可扩展点

后续新增规则时，只需：

1. 新增一个 `AutoRectificationRule` 实现
2. 在 `resolveAutoRectificationRule()` 中注册
3. 在系统设置中暴露新的规则编码

不需要重写：

1. 报告发布接口
2. 整改单创建服务
3. 系统设置持久化结构

建议下一阶段补充：
