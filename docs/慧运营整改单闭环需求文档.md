# 慧运营整改单闭环需求文档

## 1. 背景

当前 `vision-agent` 已经可以把巡检报告发布到 `report-system`，客户用户也可以在 `report-system` 的报告详情和巡检结果详情页中查看问题，并对单张结果执行“复核”操作。

当前复核链路的问题是：

1. 用户提交复核时，只是修改本地复核状态并留下备注。
2. 复核建议不会实际驱动门店去整改。
3. 页面上看不到“这次问题是否已经下发整改单、整改单是否完成整改”的闭环状态。

因此需要在 `report-system` 中新增“慧运营整改单闭环”能力：  
当用户在巡检结果详情页点击“提交复核”时，把用户勾选的问题和图片链接同步下发到慧运营，创建整改单，并在本地持续跟踪该结果对应的整改单状态。

---

## 2. 目标

本期目标：

1. 在巡检结果处理页的“复核操作”中，支持勾选问题后提交整改单。
2. 调用慧运营开放平台接口创建整改单。
3. 若勾选问题拼接后的描述超过长度限制，自动拆解为多个整改单并继续下发。
4. 新增“整改单查询”能力，查询本次巡检结果对应的整改单是否已整改完成。
5. 在“最近记录”区域展示：
   - 本次问题下发了哪些整改单
   - 每个整改单的整改状态
   - 是否已整改完成
6. 将慧运营 API 对接逻辑抽离为独立模块，便于后续迭代。

非目标：

1. 本期不改 `vision-agent`。
2. 本期不改报告发布契约。
3. 本期不实现慧运营侧整改单编辑、撤回、催办等扩展动作。
4. 本期不实现复杂的智能拆单策略，但支持基于长度上限的自动拆单。

---

## 3. 当前现状

当前系统已具备以下基础：

1. `report-system` 中已经有巡检结果详情页：
   - `/reports/{reportId}/results/{resultId}?panel=review`
2. 复核操作页已经支持：
   - 展示本结果对应的问题列表
   - 勾选问题
   - 提交复核备注
   - 修改本地结果复核状态
   - 记录本地复核日志
3. 系统设置中已经维护慧运营相关配置，存放在数据库中：
   - `uri`
   - `route`
   - `appid`
   - `secret`
   - `rateLimitCount`
   - `rateLimitWindowMs`

当前缺失的是：

1. 提交复核时没有调用慧运营 API。
2. 没有本地“整改单下发记录”数据模型。
3. 没有“整改单状态查询”任务或查询入口。
4. 最近记录中没有展示整改单闭环状态。
5. 缺少后台任务调度能力，无法按固定间隔回查未完结整改单状态。

## 4. 业务范围

### 4.1 入口页面

本次需求主要发生在：

1. 巡检结果详情页的“复核操作”Tab
2. 最近记录区域
3. 系统设置页

### 4.2 主业务流程

目标流程如下：

1. 用户进入巡检结果详情页。
2. 在“复核操作”Tab 中勾选问题。
3. 用户填写复核备注。
4. 点击“提交复核”。
5. 系统校验勾选问题拼接后的描述长度。
6. 若长度未超限：
   - 获取慧运营 token
   - 调用“新增整改单”接口
   - 本地记录整改单下发日志
   - 更新本地复核状态
7. 若长度超限：
   - 系统自动拆解为多个整改单请求
   - 每个整改单描述不超过配置上限
   - 全部下发成功后再更新本地复核完成状态
8. 后续系统查询该结果对应整改单状态。
9. 在“最近记录”中展示整改单是否已整改。

## 5. 慧运营 API 对接范围

基于你提供的 [慧运营整改单相关API.md](/Users/wick/Downloads/慧运营整改单相关API.md)，本期只接以下 3 个接口能力：

### 5.1 获取身份凭证

- `GET /sign`
- 入参：
  - `key`：一般为企业编码，文档示例为 `appid`
  - `secret`

用途：

1. 调用创建整改单前获取 token
2. 调用查询整改单列表前获取 token

### 5.2 新增整改单

- `POST /route/ri/open/item/create`
- 必要字段：
  - `storeId` 或 `storeCode`
  - `description`
  - `shouldCorrected`
  - `imageUrls`

约束：

1. `description` 最大长度 500
2. `imageUrls` 最多 9 个

### 5.3 整改单列表查询

- `POST /route/ri/open/item/list`

当前文档中可用于判断整改状态的字段包括：

1. `disqualifiedId`：整改单 ID
2. `ifCorrected`：整改状态
   - `0` 未整改
   - `1` 已整改
   - `2` 待审核
3. `storeCode`
4. `storeId`
5. `description`
6. `createTime`
7. `shouldCorrected`
8. `realCorrected`
9. `realCorrectedTime`
10. `modifiedTime`

## 6. 需求设计

## 6.1 提交复核时创建整改单

### 6.1.1 页面交互

在“复核操作”Tab 中：

1. 问题列表支持勾选。
2. 用户点击“提交复核”时：
   - 系统读取当前勾选的问题，生成整改单描述。
   - 读取当前结果图片链接

### 6.1.2 提交行为

当点击“提交复核”时，系统需要做两件事：

1. 调用慧运营新增整改单接口
2. 仅在整改单创建成功后，再把本地结果状态更新为“已复核 / completed”

推荐约束：

1. 若慧运营创建失败，则本地复核不应直接完成。
2. 这样可以保证“复核完成”与“已成功下发整改单”保持一致。

### 6.1.3 描述内容生成规则

新增整改单时，`description` 由以下内容拼接而成：

1. 用户勾选的问题标题列表，每个标题占一行，并添加连续序号。
2. 当前实现中，远端整改单 `description` 只使用勾选问题生成，不拼接本地复核备注。

推荐拼接格式：

```text
1. 商品标签未朝外（面向消费者）
2. 地面黄色瓶装啤酒堆标签未朝外
3. 货架上罐装啤酒标签未朝外

请门店于要求日期前完成整改，并上传整改结果。
```

说明：

1. `description` 的最终长度要做实际字符数校验。
2. 图片链接不拼进 `description`，而是走 `imageUrls` 字段。

### 6.1.4 长度限制校验

当前慧运营接口要求：

1. `description` 最大长度 500

本期要求：

1. `500` 不写死在代码中，改为配置项，写入慧运营 API 系统设置中。
2. 系统默认值为 `500`。
3. 当实际拼接结果超过限制时：
   - 自动拆解为多个整改单描述
   - 每个整改单描述不超过 `500` 字
   - 拆解后每个整改单描述中的问题序号保持连续

推荐后续新增系统设置项：

1. `rectificationDescriptionMaxLength`

### 6.1.5 图片链接规则

提交整改单时，`imageUrls` 使用当前巡检结果关联的图片公网 URL。

本期建议规则：

1. 默认只上传当前结果页对应的主图片 URL。
2. 若未来一条结果支持多个图片，再扩展为最多 9 张。

## 6.2 整改单查询与闭环状态

### 6.2.1 查询目标

系统需要能回答这个问题：

“这个门店的这一次巡检结果，对应的整改单是否已经整改完成？”

这不是泛化的“门店全部整改单列表”，而是要聚焦到：

1. 当前 `reportId`
2. 当前 `resultId`
3. 当前门店
4. 当前这次用户下发的整改单
5. 后续整改单会形成一个独立功能，每个整改单都有自己的数据记录（这可能影响你的数据库设计，需要预埋一个“整改单记录表”：下发的整改单和回查的整改单状态）。

### 6.2.2 本地追踪原则

不能只依赖页面打开时临时去慧运营查一次，因为这样会丢失闭环上下文。

因此本期建议：

1. 本地新增“整改单下发记录表”
2. 每次创建整改单成功后，都落本地记录
3. 后续查询状态时，基于本地下发记录去回查慧运营

### 6.2.3 建议新增本地表

建议新增例如：

1. `report_rectification_order`

建议字段至少包括：

1. `id`
2. `report_id`
3. `result_id`
4. `store_id`
5. `store_code`
6. `store_name`
7. `source_review_log_id`
8. `huiyunying_order_id`
9. `request_description`
10. `selected_issues_json`
11. `image_urls_json`
12. `request_payload_json`
13. `response_payload_json`
14. `status`
15. `if_corrected`
16. `should_corrected`
17. `real_corrected_time`
18. `last_synced_at`
19. `created_by`
20. `created_at`
21. `updated_at`

说明：

1. `status` 是本地集成状态，例如：
   - `created`
   - `sync_failed`
   - `corrected`
   - `pending_review`
2. `if_corrected` 保存慧运营原始状态：
   - `0`
   - `1`
   - `2`

### 6.2.4 查询策略

建议支持两种查询方式：

1. 被动查询
   - 用户打开结果详情页时，如果该结果存在整改单记录，则回查最新状态
2. 主动同步
   - 通过后台任务调度管理器，定时刷新未完成整改单状态（例如每 10 分钟刷新一次）

本期要求：

1. 支持“打开页面时按需查询”
2. 支持后台定时同步未完结整改单状态
3. 后台查询间隔放在系统设置中维护，配置项建议为 `rectificationSyncIntervalMs`
4. 当查询间隔配置为 `0` 时，表示关闭后台自动同步，仅保留页面按需刷新

## 6.3 最近记录展示要求

当前“最近记录”主要展示本地复核日志。  
本期改造后，需要在最近记录中同时看到“复核动作”和“整改单闭环状态”。

### 6.3.1 展示内容

针对当前结果，需要展示：

1. 哪次复核动作触发了整改单创建
2. 下发了哪些整改单
3. 每个整改单当前状态
4. 是否整改完成
5. 若已整改，显示整改完成时间

### 6.3.2 推荐展示字段

每条整改单记录建议展示：

1. 整改单 ID
2. 创建时间
3. 要求整改日期
4. 选中的问题摘要
5. 当前整改状态
   - 未整改
   - 已整改
   - 待审核
6. 最近同步时间

### 6.3.3 展示位置

建议在“最近记录”区域中按时间倒序展示：

1. 复核日志
2. 复核日志下关联的整改单状态摘要

或者拆成两个块：

1. 最近复核记录
2. 整改单闭环记录

本期推荐第二种，更清晰。

## 6.4 慧运营 API 抽象层要求

你已经明确要求：  
“对接慧运营 API，需要抽象成单独文件处理，以便后续 API 迭代。”

本期必须按这个原则实现，不能把 HTTP 调用逻辑写在 route 或 report service 里。

建议结构：

```text
backend/
  integrations/
    huiyunying/
      huiyunying.types.ts
      huiyunying-client.ts
      huiyunying-auth.ts
      huiyunying-rectification.service.ts
      huiyunying-rectification.mapper.ts
```

职责建议：

1. `huiyunying-client.ts`
   - 封装基础 HTTP 请求
   - 封装限流
   - 封装错误处理

2. `huiyunying-auth.ts`
   - 负责 `/sign` 获取 token

3. `huiyunying-rectification.service.ts`
   - 创建整改单
   - 查询整改单列表

4. `huiyunying-rectification.mapper.ts`
   - 负责本地数据与慧运营 API 请求/响应的字段映射

要求：

1. 页面和 route 不直接拼慧运营接口 URL。
2. route 只调用本地业务 service。
3. 若后续慧运营 API 字段变化，只改集成层。

## 8. 失败处理要求

### 8.1 创建整改单失败

当点击“提交复核”时，如果创建整改单失败：

1. 不应把本地结果直接改成“已复核”
2. 页面需要给出明确失败提示
3. 失败原因要记录到本地日志或整改单记录表中

### 8.2 查询失败

如果整改单查询失败：

1. 不影响结果详情页基本展示
2. 页面上显示“整改单状态获取失败，请稍后重试”
3. 保留上次成功同步状态

### 8.3 限流控制

对慧运营调用必须使用当前系统设置中的限流参数：

1. `rateLimitCount`
2. `rateLimitWindowMs`

## 9. 数据流设计

### 9.1 提交复核

```text
用户勾选问题
-> 点击提交复核
-> 拼接 description
-> 校验长度
-> 获取慧运营 token
-> 调用新增整改单
-> 写本地整改单记录
-> 写本地复核日志
-> 更新结果状态 completed
```

### 9.2 页面查看

```text
打开结果详情页
-> 加载本地复核日志
-> 加载本地整改单记录
-> 若存在未完成整改单
-> 调用慧运营查询接口刷新状态
-> 更新本地整改单状态
-> 页面展示最新闭环情况
```

1. 用户在“复核操作”Tab 中勾选问题并点击“提交复核”时，会实际调用慧运营新增整改单接口。
