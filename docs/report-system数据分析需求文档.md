# Report System 数据分析需求文档

## 1. 背景

`report-system` 当前已经具备以下业务能力：

- 接收 `vision-agent` 发布的巡检报告
- 按批次、门店、结果、问题项、inspection 明细落盘
- 支持人工复核、本地复核日志、整改单闭环记录
- 已形成统一的巡检结果语义：
  - `issue_found`
  - `pass`
  - `inconclusive`
  - `inspection_failed`

随着业务推进，系统不仅需要“查看某一批报告和逐条处理问题”，还需要基于已落盘的业务数据形成管理分析能力，支撑：

- 巡检质量分析
- 问题分布分析
- 人工复核效率分析
- 整改闭环效率分析
- 组织 / 门店治理分析

因此需要在现有系统中新增“数据分析”能力，并明确指标口径、数据边界、实现分层与迭代计划。

---

## 2. 目标

本次数据分析建设目标如下：

1. 基于 `report-system` 已落盘的业务数据，建立稳定的分析口径。
2. 新增独立的 `analytics` 查询层，不将分析逻辑散落到现有报告业务查询中。
3. 提供一版面向运营和管理者的数据分析页面。
4. 确保分析结果与当前巡检语义、复核逻辑、整改单闭环逻辑保持一致。
5. 保持权限收口，分析结果只展示当前用户授权范围内的数据。
6. 支持从 `门店 / 加盟商 / 运营组织` 三类主体审视巡检质量与治理动作。
7. 在统计型指标之外，逐步补齐能够驱动改善动作的治理型指标。

---

## 3. 非目标

本期不包含以下内容：

- 不引入独立 BI 平台
- 不引入 PostgreSQL / ClickHouse / OLAP 引擎
- 不做实时流式分析
- 不做复杂预测类算法分析
- 不做多租户跨企业聚合分析
- 不做高度自由的自定义报表设计器

本期目标是：在当前单体系统与 SQLite 落盘数据基础上，完成一版稳定、可维护、可迭代的数据分析能力。

---

## 4. 数据基础与分析边界

当前可用于分析的核心表如下：

- `report`
  - 批次级数据
  - 适合做时间、计划、主题、企业维度分析

- `report_store`
  - 门店级快照
  - 适合做组织、门店、门店结果汇总

- `report_image`
  - 单次巡检结果
  - 适合做巡检结论分布、结果处理、图片级复核分析

- `report_issue`
  - 问题项明细
  - 适合做问题类型、技能命中、严重程度、问题排行

- `report_inspection`
  - inspection 过程明细
  - 适合做“无法判定 / 巡检失败”的语义辅助分析

- `report_review_log`
  - 人工复核日志
  - 适合做复核效率、人工处理量、状态变更分析

- `report_rectification_order`
  - 整改单闭环记录
  - 适合做整改下发量、整改完成率、超期分析、闭环效率分析

- `report_rectification_sync_batch`
  - 整改单状态同步批次
  - 适合做同步任务质量监控

- `report_rectification_sync_log`
  - 单据同步日志
  - 适合做同步异常、状态变化追踪

### 分析边界

本次分析仅基于 `report-system` 已落盘的数据进行，不直接依赖 `vision-agent` 在线查询。

也就是说：

- 分析结果以 `report-system` 当前库中数据为准
- 历史批次是否纳入统计，以本地是否已落盘为准
- 整改闭环状态以本地同步回写后的整改单记录为准

## 5. 核心分析口径

### 5.1 巡检结果语义口径

系统必须统一使用当前已有的结果语义，不允许回退为“问题数是否为 0”这种粗口径。

#### 巡检结果语义定义

- `issue_found`
  - 当前结果存在问题项

- `pass`
  - inspection 成功且未发现问题

- `inconclusive`
  - 无问题项，但 inspection 无法形成明确结论
  - 例如目标缺失、画面异常、低置信度

- `inspection_failed`
  - inspection 失败或异常

### 5.2 复核口径

- `pass`
  - 批次入库时即自动标记为 `已复核`

- `issue_found`
  - 需要人工处理
  - 处理完成后可进入整改单闭环

- `inconclusive`
  - 仅支持本地复核
  - 不进入整改单下发

- `inspection_failed`
  - 仅支持本地复核
  - 不进入整改单下发

### 5.3 整改闭环口径

整改单分析口径以 `report_rectification_order` 为准，建议统一划分为：

- 已下发
- 待整改
- 已整改
- 超期未整改
- 同步异常

其中：

- `已下发`
  - 本地已创建整改单记录

- `待整改`
  - 已下发但尚未完成整改

- `已整改`
  - 本地同步结果显示整改完成

- `超期未整改`
  - 当前日期超过 `should_corrected` 且未完成整改

- `同步异常`
  - 整改单同步过程中出现失败或业务异常

## 6. 维度设计

首期分析建议支持以下维度：

- 时间维度
  - 日
  - 周
  - 时间范围

- 组织维度
  - 企业
  - 组织
  - 加盟商
  - 门店

- 业务维度
  - 报告类型
  - 报告主题
  - 巡检计划
  - 技能
  - 问题类型
  - 巡检结论
  - 复核状态
  - 整改状态

### 6.1 责任主体维度

除统计维度外，分析模块还需要逐步支持以下责任主体维度：

- 门店
  - 问题处理和整改执行主体

- 加盟商
  - 对名下门店经营质量和整改质量负责

- 运营组织
  - 对辖区内门店和加盟商的治理推进、督导与闭环效率负责

后续 analytics 设计中，必须逐步区分：

- 问题发生在哪
- 问题应该由谁推动整改

- 待复核数

## 8. 指标明细定义

### 8.1 巡检规模指标

- `report_count`
  - 统计时间范围内已发布批次数

- `store_count`
  - 统计时间范围内去重门店数

- `result_count`
  - 统计时间范围内巡检结果总数

- `issue_count`
  - 统计时间范围内问题项总数

### 8.2 巡检结果指标

- `issue_found_count`
  - `issue_found` 结果数

- `pass_count`
  - `pass` 结果数

- `inconclusive_count`
  - `inconclusive` 结果数

- `inspection_failed_count`
  - `inspection_failed` 结果数

### 8.3 复核指标

- `pending_review_count`
  - 当前待复核结果数

- `completed_review_count`
  - 当前已复核结果数

- `auto_completed_review_count`
  - 批次入库时自动复核完成的结果数

- `manual_completed_review_count`
  - 通过人工复核完成的结果数

- `average_review_duration`
  - 从结果入库到首次完成复核的平均耗时

### 8.4 整改单指标

- `rectification_order_count`
  - 已下发整改单总数

- `rectification_completed_count`
  - 已整改完成整改单数

- `rectification_pending_count`
  - 尚未整改完成整改单数

- `rectification_overdue_count`
  - 超过整改截止日期仍未完成整改的整改单数

- `rectification_close_rate`
  - 已整改数 / 已下发数

- `average_rectification_duration`
  - 从整改单创建到整改完成的平均耗时

## 9. 权限要求

分析页必须沿用当前系统权限范围。

### 9.1 管理员

- 可查看当前租户范围内全部分析数据

### 9.2 普通查看者

- 只能查看其授权组织范围内的数据分析结果

### 9.3 复检员

- 只能查看其授权组织范围内的数据分析结果

### 9.4 权限收口原则

所有分析查询都必须基于当前 `RequestContext` 做数据范围过滤，不允许：

- 全局汇总结果泄露给受限用户
- 列表受限但概览卡片全量
- 排行列表超出授权组织范围

## 10. 技术方案

### 10.1 分层原则

不建议把分析逻辑直接写进当前的 `report.service` 或页面层。

建议新增独立分析模块：

```text
backend/
  analytics/
    analytics.types.ts
    analytics.repository.ts
    sqlite-analytics.repository.ts
    analytics.service.ts
    analytics.module.ts
```

### 10.2 查询层职责

- `analytics.repository`
  - 负责聚合 SQL 查询

- `analytics.service`
  - 负责指标编排、参数校验、默认口径、权限上下文收口

- 页面层
  - 只读取已整理好的 view model
  - 不直接拼分析 SQL

### 10.3 实现建议

一期仍基于 SQLite 实现分析查询。

建议：

- 聚合查询优先在 SQL 层完成
- 不用页面层循环全量数据后再统计
- 对高频查询使用明确筛选条件和索引

## 13. 数据口径风险

以下风险必须在实现时明确处理：

### 13.1 旧数据口径差异

历史数据可能在结果语义字段、自动复核策略、整改单闭环逻辑上存在迭代差异，需要确认是否需要兼容旧批次。

### 13.2 同步延迟

整改单状态分析依赖同步任务，分析页展示的是“本地最近同步状态”，不保证与慧运营完全实时一致。

### 13.3 自动复核影响

`pass` 结果在入库时即自动复核，分析时需要明确区分自动复核与人工复核，否则复核效率分析会失真。

### 13.4 结果语义不能退化

分析实现必须统一复用现有语义分类规则，不能再次回退为：

- 问题数 > 0
- 问题数 = 0

这种过于粗糙的统计方式。

## 14. 验收标准

完成一期分析能力后，应满足：

1. 系统新增 `/analytics` 一级页面
2. 页面支持按时间与组织范围筛选
3. 能展示巡检规模、巡检结论分布、问题排行、复核效率、整改闭环核心指标
4. 指标口径与当前巡检结果语义一致
5. 普通查看者与复检员只能看到授权范围内数据
6. 分析页核心查询响应可接受，不依赖前端全量遍历统计

## 16. 实施优先级建议

建议按以下优先级推进：

1. 先做分析模块与指标口径
2. 再做概览页
3. 再做趋势与排行
4. 最后做导出与联动

不建议先做复杂 UI，再回头补指标口径。
