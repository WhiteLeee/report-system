# Report System 数据分析页改造实施方案

## 1. 文档目的

本文档用于将 `report-system` 当前的数据分析页改造目标，细化为可 review、可排期、可实施的页面改造任务清单。

本文档基于以下三份已确认文档：

- [report-system数据分析需求文档.md](/Users/wick/Desktop/ZhiQing%20Selection/SourceCode/report-system/docs/report-system数据分析需求文档.md)
- [report-system数据分析技术架构.md](/Users/wick/Desktop/ZhiQing%20Selection/SourceCode/report-system/docs/report-system数据分析技术架构.md)
- [report-system多角色治理分析设计方案.md](/Users/wick/Desktop/ZhiQing%20Selection/SourceCode/report-system/docs/report-system多角色治理分析设计方案.md)

本文档不重新定义分析底层口径，重点解决：

- `/analytics` 页面如何更符合管理分析首页定位
- 页面展示结构如何与分析需求和技术架构对齐
- 现有模块哪些保留、哪些调整、哪些迁移
- 如何拆成可执行的实施任务

## 2. 当前问题回顾

当前 `/analytics` 已经具备以下能力：

- 概览卡片
- 巡检结论分布
- 近 14 日趋势
- 问题类型排行
- 组织排行
- 门店排行
- 复核效率
- 整改超期排行
- CSV 导出
- 分析筛选

当前页面虽然“能统计”，但距离“管理驾驶舱首页”还有明显差距，主要问题如下：

### 2.1 信息层级不够清楚

当前概览、趋势、排行、复核、整改信息都堆在一个页面中，阅读路径偏散，用户需要自己理解区块关系。

### 2.2 缺少管理问题导向

当前页面更像“把已有统计展示出来”，而不是直接回答管理者最关心的问题：

- 最近巡检质量怎么样
- 问题集中在哪
- 人工复核是否有积压
- 整改闭环是否卡住
- 哪些组织和门店最值得关注

### 2.3 技术信息与业务信息边界需要更明确

`/analytics` 应是业务分析首页，不应混入：

- facts / snapshots 这类底层术语
- 任务执行状态
- 调度健康状态

这些信息应留在：

- `/analytics/jobs`
- `系统设置 > 分析任务设置`

### 2.4 筛选区当前仍偏重

虽然已经改为默认折叠，但仍然是“完整筛选表单”。缺少：

- 当前筛选摘要
- 高频筛选与高级筛选的分层

### 2.5 指标表达仍需业务化

当前部分字段和展示仍偏技术口径，例如：

- `result_count`
- `pending_review_count`
- `rectification_close_rate`

这些需要统一成面向运营和管理者的业务表达。

### 2.6 缺少加盟商与责任视角

当前页面重点覆盖的是：

- 组织
- 门店
- 趋势
- 排行

但还没有把 `加盟商` 作为正式分析主体纳入首页结构，也没有区分：

- 问题发生对象
- 改善责任对象

这会让页面更像统计页，而不是治理首页。

## 3. 改造目标

本轮数据分析页改造目标如下：

1. 将 `/analytics` 从“统计页”升级为“管理分析首页”。
2. 建立清晰的信息架构，按管理问题组织页面，而不是按数据字段堆模块。
3. 保持底层分析口径不变，不推翻当前 analytics 架构。
4. 将技术运维信息完全移出分析首页。
5. 为后续 drill-down、导出和 snapshot 扩展保留稳定页面骨架。
6. 为后续接入加盟商视角、责任归属视角和治理型指标预留页面结构。

---

## 4. 页面定位

`/analytics` 的正式定位定义为：

`巡检数据分析与治理首页`

也就是说：

- 它不是报告页
- 不是导出页
- 不是技术任务页
- 而是面向运营、管理者和治理人员的分析首页

其主要职责是：

1. 看整体情况
2. 看质量变化
3. 看问题集中区域
4. 看复核和整改堵点
5. 快速定位需要跟进的组织和门店
6. 为后续定位需要重点跟进的加盟商预留稳定入口

---

## 5. 页面信息架构

改造后，页面建议按以下顺序组织。

### 5.1 筛选摘要区

位置：页面最上方，位于概览卡片之前。

职责：

- 默认折叠高级筛选
- 展示当前筛选摘要
- 允许展开并修改筛选条件

展示内容：

- 当前时间范围
- 当前企业
- 当前组织
- 当前报告类型
- 当前是否使用高级筛选

说明：

- 页面默认不直接展示完整筛选表单
- 如果用户已经带筛选条件进入，则自动展开

### 5.2 经营概览区

位置：第一屏核心区域

职责：

用 6 到 8 张卡片回答“当前总体情况”。

建议卡片：

- 巡检批次数
- 巡检门店数
- 巡检图片数
- 问题总数
- 待人工复核
- 整改单总数
- 已整改数
- 整改闭环率

要求：

- 卡片名称统一业务化
- 每张卡片保留副文案
- 卡片顺序按业务优先级排布

### 5.3 巡检质量区

职责：

回答“巡检本身质量如何”。

建议模块：

- 巡检结论分布
- 近 14 日趋势

建议重点：

- 突出 `发现问题 / 未发现问题 / 无法判定 / 巡检失败`
- 让用户能快速发现：
  - 无法判定是否升高
  - 巡检失败是否增多
  - 问题结果是否持续增加

### 5.4 问题分析区

职责：

回答“主要问题是什么”。

建议模块：

- 问题类型 Top N
- 技能命中分布
- 严重程度分布

说明：

当前实现要求：

- 问题类型继续保留表格，负责责任定位与下钻
- 技能命中分布优先使用图表，负责识别规则和风险集中点
- 严重程度分布优先使用图表，负责识别高优先级问题占比

### 5.5 复核与整改区

职责：

回答“治理动作是否跟得上”。

建议合并为一个大区，而不是分散展示。

建议模块：

- 待人工复核
- 自动已复核
- 人工已复核
- 平均复核时长
- 整改单状态分布
- 超期未整改数
- 整改闭环率

说明：

该区要突出“积压”和“堵点”，而不只是展示静态数字。

### 5.6 组织与门店定位区

职责：

回答“问题集中在哪”。

建议模块：

- 组织问题排行
- 门店问题排行
- 超期整改排行
- 待复核积压排行

说明：

这一块是管理动作的直接入口，应支持后续 drill-down。

### 5.7 加盟商治理预留区

本轮可以暂不完整落地加盟商图表，但页面结构必须预留：

- 加盟商问题排行
- 加盟商闭环率排行
- 高风险加盟商识别

原因：

- 多角色治理分析已经明确把加盟商视为正式责任主体
- 后续如果首页完全没有该区块，将导致页面再次重构

## 6. 页面模块调整策略

### 6.1 保留模块

以下模块继续保留：

- 概览卡片
- 巡检结论分布
- 近 14 日趋势
- 问题类型排行
- 组织排行
- 门店排行
- 复核效率
- 整改超期排行
- 导出能力

### 6.1.1 后续扩展时的角色优先级

在后续页面扩展中，建议按以下优先级补角色视角：

1. 运营组织
2. 加盟商
3. 门店

原因：

- 门店明细与处理流程当前已在报告详情中较完整
- analytics 首页更应优先解决督导和经营责任视角

### 6.2 调整模块

以下模块保留数据能力，但调整展示位置和表达：

- 分析筛选
  - 从完整表单改为“筛选摘要 + 折叠高级筛选”

- 复核效率
  - 从独立零散块调整到“复核与整改区”

- 整改超期排行
  - 从孤立模块调整到“组织与门店定位区”

### 6.3 从首页移除的内容

以下内容不再出现在 `/analytics` 首页：

- `result_fact_rebuild`
- `daily_snapshot_rebuild`
- health status
- checkpoint
- facts / snapshots 术语
- 技术任务执行情况

这些内容只保留在：

- `/analytics/jobs`
- `系统设置 > 分析任务设置`

## 7. 指标与文案统一规范

页面必须统一使用以下业务文案。

### 7.1 总览指标

- `report_count` -> `巡检批次数`
- `store_count` -> `巡检门店数`
- `result_count` -> `巡检图片数`
- `issue_count` -> `问题总数`
- `pending_review_count` -> `待人工复核`
- `auto_completed_review_count` -> `自动已复核`
- `rectification_order_count` -> `整改单总数`
- `rectification_close_rate` -> `整改闭环率`

### 7.2 结果语义

- `issue_found` -> `发现问题`
- `pass` -> `未发现问题`
- `inconclusive` -> `无法判定`
- `inspection_failed` -> `巡检失败`

### 7.3 状态表达要求

- 页面上不直接展示技术状态码
- 页面上不直接展示 repository / facts / snapshot 等术语
- 所有状态优先用中文业务标签

---

## 8. 筛选区改造要求

### 8.1 高优先级筛选

默认在筛选摘要和展开后优先展示：

- 时间范围
- 企业
- 组织
- 报告类型

### 8.2 高级筛选

折叠后展示：

- 门店
- 报告主题
- 计划 ID

### 8.3 行为要求

- 默认折叠
- 已带筛选参数时自动展开
- 页面顶部显示当前筛选摘要
- 支持一键重置

---

## 9. 技术实现约束

本轮改造只允许发生在展示层和 analytics view model 层，不允许破坏已落地架构。

必须遵守：

1. 页面不直接拼业务表统计
2. 页面继续通过 analytics service 取数
3. 页面不重新定义结果语义
4. 技术运维状态不回流到分析首页
5. 后续 snapshot 扩展时，页面骨架无需重写

---

## 10. 实施拆解

本轮建议拆成 4 个实施阶段。

### 阶段一：结构重排

目标：

先不加新数据，只重组 `/analytics` 页面区块顺序和展示分组。

任务：

1. 将页面重组为：
   - 筛选摘要区
   - 经营概览区
   - 巡检质量区
   - 问题分析区
   - 复核与整改区
   - 组织与门店定位区

2. 将现有模块映射进上述区块

3. 将技术任务相关内容彻底移出分析首页

验收标准：

- 页面区块顺序符合本方案
- 首页不再出现技术任务信息

### 阶段二：文案与业务表达统一

目标：

统一所有标题、指标名、说明文案。

任务：

1. 替换所有技术字段名为业务名称
2. 替换所有结果语义的内部代号为中文业务文案
3. 统一各区块标题和副文案风格

验收标准：

- 页面不存在技术字段名直接暴露
- 页面语言符合业务分析首页定位

### 阶段三：筛选区重构

目标：

让筛选区从“表单”变成“摘要 + 高级筛选”。

任务：

1. 新增筛选摘要条
2. 保持默认折叠
3. 已带参数时自动展开
4. 将高频筛选与高级筛选区分展示

验收标准：

- 页面初始阅读不被完整筛选区打断
- 用户可快速理解当前筛选口径

### 阶段四：行动定位与联动优化

目标：

增强页面的行动导向。

任务：

1. 将组织排行、门店排行、超期排行统一成“定位区”
2. 为排行项补充 drill-down 入口
3. 将导出和 drill-down 保持一致口径

验收标准：

- 排行区具备明确行动导向
- 用户可从分析定位到具体业务页

---

## 11. 本轮不做的事项

以下事项不属于本轮数据分析页改造范围：

- 不新增新的底层分析表结构
- 不改 analytics facts / snapshots 架构
- 不引入新数据库或 BI 工具
- 图表渲染统一使用 `shadcn-ui + recharts`
- 不引入实时流式分析
- 不扩展新的权限模型

---

## 12. 改造后预期结果

完成后，`/analytics` 应达到以下效果：

1. 页面第一眼即可看懂，而不是需要阅读多个零散统计模块
2. 管理者能够快速回答：
   - 巡检质量怎么样
   - 问题在哪里
   - 复核是否积压
   - 整改是否卡住
   - 哪些组织和门店优先处理
3. 页面结构与底层 analytics 架构一致
4. 后续继续扩展 snapshot、导出和 drill-down 时，不需要推翻页面骨架

---

## 13. 建议实施顺序

建议按如下顺序落地：

1. 阶段一：结构重排
2. 阶段二：文案统一
3. 阶段三：筛选区重构
4. 阶段四：行动定位优化

即：

先重组页面，再统一表达，再优化筛选，最后补动作联动。
