# report-system 用户管理与权限重构需求文档

## 1. 文档信息
- 文档版本：v1.1
- 创建日期：2026-04-08
- 适用范围：report-system（单企业单实例）
- 目标读者：产品、后端、前端、测试、实施交付

## 2. 背景与目标

当前项目已具备基础用户管理能力（用户列表、创建、编辑角色、状态、密码、范围授权），并已将“用户管理”从系统管理中拆为顶级导航。  
下一阶段目标是将用户管理与权限控制升级为可交付、可审计、可扩展的标准化能力，支持后续客户化交付和规则演进。

本次重构总体目标：
- 建立统一的权限模型（RBAC + Scope + Policy 扩展位）。
- 明确“平台管理能力”与“客户可见能力”边界，避免向客户暴露内部 admin。
- 形成可审计、可追踪、可回滚的权限治理闭环。
- 强化数据安全基线（认证、授权、会话、审计、敏感数据保护）。
- 与业务系统尽量解耦，采用能力边界与适配层集成。
- 为后续“自动化流程权限规则”预留扩展点。

## 3. 现状评估（基于已完成模块）

### 3.1 已完成能力
- 认证会话：用户名/密码登录，Cookie 会话，7天有效期。
- 权限检查：服务端 `requirePermission` / `hasPermission` 拦截页面与 API。
- 角色模型：`admin`、`reviewer`、`viewer`。
- 权限点：`report:read`、`review:write`、`user:manage`。
- 范围授权：`enterprise / organization / store` 三层 scope。
- 用户管理 UI：用户列表，新增/编辑弹窗，角色/状态/密码/范围管理。
- 管理入口：用户管理已作为顶部平级菜单存在。

### 3.2 现状问题
- 角色与权限颗粒较粗，仅 3 个权限点，无法支撑细分业务权限。
- 缺少权限变更审计日志（谁在何时改了什么）。
- 缺少安全策略闭环（密码复杂度/轮换、登录失败策略、会话撤销策略）。
- 角色配置能力存在后端接口，但无完整前端治理页与流程约束。
- 缺少“交付模式”控制，尚未形成“不向客户暴露平台 admin”的系统级策略。

## 4. 重构原则

1. 最小权限：默认无权限，仅授予完成职责所需最小集合。  
2. 默认拒绝：未明确允许即拒绝。  
3. 职责分离：关键能力（用户与策略管理）与业务操作分离。  
4. 服务端强校验：所有权限判断在后端完成，前端仅做可见性优化。  
5. 范围先于动作：先判断数据范围，再判断动作权限。  
6. 全量审计：权限与用户关键操作必须留痕。  
7. 可扩展：保留策略引擎扩展点，支持未来规则组合。
8. 安全优先：认证授权链路和敏感数据默认按高风险资产治理。
9. 业务解耦：用户权限域不依赖具体业务表结构，统一通过能力接口接入。

## 5. 目标能力范围

### 5.1 本阶段（必须完成）
- 用户全生命周期：创建、启停、重置密码、角色授权、范围授权。
- 角色治理：角色定义、角色权限矩阵可配置（含保护规则）。
- 菜单与接口联动：菜单可见性与接口权限保持一致。
- 审计日志：记录所有用户与权限变更行为。
- 交付模式：客户环境不暴露“平台 admin”。
- 数据安全：落实密码策略、会话治理、敏感字段保护、操作审计不可篡改约束。
- 低耦合集成：权限服务仅依赖统一资源访问接口，不直接耦合报告/整改单/分析内部表。

### 5.2 下阶段（预留）
- 策略规则（ABAC/条件权限）扩展。

## 6. 角色与权限模型设计

### 6.1 角色分层（目标）
- `platform_admin`（内部平台，仅实施/运维可见，不对客户暴露）
- `tenant_admin`（客户管理员，可管本实例用户与权限）
- `ops_manager`（运营经理，报告与复核管理）
- `reviewer`（复核执行）
- `viewer`（只读）

说明：当前项目单实例单企业，`tenant_admin` 的范围上限即当前实例全部数据。

### 6.2 权限域（目标拆分）
- 报告域：`report:read`、`report:export`
- 复核域：`review:read`、`review:write`
- 整改单域：`rectification:read`、`rectification:write`
- 分析域：`analytics:read`、`analytics:export`、`analytics:job:manage`
- 主数据域：`master-data:read`
- 用户与权限域：`user:read`、`user:write`、`role:read`、`role:write`、`scope:write`
- 系统设置域：`system:settings:read`、`system:settings:write`

### 6.3 范围模型
- `enterprise_scope`（本项目保留兼容，单实例默认当前企业）
- `organization_scope`
- `store_scope`

规则：
- 范围取交集：用户范围 ∩ 请求范围。
- 空范围表示“全部可见”仅对具备对应动作权限的角色生效。
- 组织选择应联动门店范围。

## 7. 功能需求

### 7.1 用户管理（顶级菜单）
- 用户列表：账号、显示名、状态、角色、权限摘要、范围摘要、更新时间。
- 新增用户：账号、显示名、初始密码、角色、范围（组织与门店联动选择）。
- 编辑用户：分区操作（角色、范围、状态、密码），每个操作独立提交。
- 禁用/启用：禁用后立即失效；可配置“是否踢出当前会话”。
- 自助信息：用户可查看自身账号信息（非管理页）。

### 7.2 角色与权限管理
- 角色列表：系统角色与自定义角色区分展示。
- 权限矩阵：按权限域分组勾选。
- 保护策略：
  - `platform_admin` 不可在客户模式中展示/编辑。
  - 至少保留 1 个可管理用户与权限的管理员角色。
  - 不允许删除系统保留角色。

### 7.3 范围授权
- 范围选择组件：组织树 + 联动门店多选。
- 支持“按组织全选”与“单店细化”。
- 存储为结构化 scope 记录，避免文本拼接输入。

### 7.4 交付模式与账号可见性
- 增加“交付模式”系统设置（`internal` / `customer`）。
- `customer` 模式下：
  - 隐藏 `platform_admin` 账号与相关菜单。
  - 禁止对 `platform_admin` 的任何读写 API。
- `internal` 模式下：完整可见。

### 7.5 审计日志
- 审计事件覆盖：
  - 用户创建/禁用/启用/密码重置
  - 角色变更
  - 权限矩阵变更
  - 范围变更
  - 系统交付模式变更
- 记录字段：
  - 操作人、目标对象、操作类型、变更前、变更后、时间、来源IP、请求ID。

## 8. 非功能需求

### 8.1 安全
- 密码最小强度：长度、复杂度可配置（默认至少 12 位 + 多字符集）。
- 登录失败保护：失败次数阈值、临时锁定策略。
- 会话安全：支持手动失效（登出全部设备）。
- 敏感字段保护：密码仅哈希存储；密钥类配置仅密文或受控存储；日志脱敏展示。
- 权限防绕过：所有管理接口必须二次校验（权限 + 角色保护规则 + 数据范围）。
- 审计防抵赖：审计日志不可被普通管理角色删除或修改。
- 高风险操作确认：角色降权、停用管理员、批量授权需二次确认与审计记录。

### 8.2 可维护性
- 权限点统一定义，前后端共享枚举与描述。
- 权限校验中间件统一收敛，避免散落重复判断。
- 新增权限点不应要求大范围改表结构。

### 8.3 性能
- 用户列表、角色列表、审计列表支持分页与条件检索。
- 权限判断应使用缓存/预加载策略，避免每请求重复多表查询。

### 8.4 解耦与架构约束
- 边界划分：`auth-domain`（用户/角色/权限/范围/审计）与 `business-domain`（报告、整改单、分析）分离。
- 集成方式：业务系统只消费标准权限能力（如 `authorize(resource, action, scope)`），不直接读取用户角色表。
- 资源映射：由权限域维护“资源类型/动作”字典，业务域仅上报资源标识，不反向依赖权限存储结构。
- 适配层原则：对外部系统（如慧运营）通过 adapter 调用，不将外部字段模型渗透到权限核心模型。
- 演进兼容：新增业务模块时只扩展资源动作与策略，不修改既有用户核心表结构。

## 9. 数据与接口需求

### 9.1 数据层（建议新增）
- `auth_audit_log`：权限与用户操作审计表。
- `auth_setting` 或复用 `system_setting`：保存交付模式与安全策略。
- 现有 `report_user / role / permission / user_scope` 保留并演进。
- `auth_security_policy`（可选）：密码策略、会话策略、登录失败策略参数化配置。
- 审计日志字段需包含 `request_id`、`ip`、`user_agent`、`before_json`、`after_json`。

### 9.2 接口层（目标）
- 用户接口：列表、创建、更新、状态变更、密码重置、会话失效。
- 角色接口：列表、权限矩阵读取/更新。
- 审计接口：分页查询、按对象/操作人/时间筛选。
- 所有管理接口需 `user:write/role:write/scope:write` 细粒度权限。
- 权限判定接口统一化：提供内部服务接口（或模块函数）给业务域调用，禁止业务域拼装权限 SQL。

## 10. 页面与交互要求

### 10.1 导航
- 顶级菜单：报告列表、整改单、数据分析、用户管理、系统管理。
- 用户管理与系统管理互相平级，不再从属。

### 10.2 用户管理页面
- 结构：列表主视图 + 弹窗操作（新增/编辑）。
- 编辑弹窗分区：基础信息、角色权限、范围授权、安全操作。
- 风格要求：统一 shadcn/ui 组件体系。

### 10.3 角色权限页面（新增）
- 角色列表 + 权限矩阵 + 变更确认弹窗。
- 高风险变更（取消管理权限）需二次确认。

### 10.4 审计页面（新增）
- 可筛选的审计流水列表。
- 支持查看变更 diff（before/after）。

## 11. 验收标准（DoD）

1. 客户模式下不可看到/操作平台 admin。  
2. 用户管理支持完整生命周期操作，且全部有审计记录。  
3. 角色权限变更可视化且可回溯。  
4. 组织与门店范围联动选择可用，后端权限生效一致。  
5. 所有受保护页面与 API 均由服务端权限校验拦截。  
6. 关键流程（新增用户、改角色、改范围、禁用用户）具备自动化测试覆盖。  
7. 敏感信息不明文落库、不在审计明文外泄（日志脱敏检查通过）。  
8. 权限模块不直接依赖报告/整改单业务表；新增业务资源仅通过资源动作映射扩展。  

## 12. 分阶段实施建议

### Phase 1（基础治理）
- 完成角色/权限点拆分与接口收敛。
- 完成交付模式开关与 admin 隐藏策略。
- 完成审计日志落库与查询。

### Phase 2（管理体验）
- 上线角色权限管理页与审计页面。
- 完善用户管理弹窗分区与联动授权组件。

### Phase 3（安全增强）
- 登录失败策略、密码策略、会话撤销能力上线。
- 补齐权限与审计自动化回归测试。

## 13. 非目标（本次不做）
- 多租户同库并行管理。
- 外部 IAM/SSO 接入。
- 审批流引擎与流程编排。

## 14. 风险与依赖

- 角色与权限点拆分会影响既有菜单可见性和接口保护，需同步改造与回归测试。
- 若历史用户数据存在脏范围（无效组织/门店编码），需在迁移时做清洗。
- 交付模式切换需避免误锁死管理入口（需保底管理员恢复机制）。
