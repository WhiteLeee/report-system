# Report System 独立项目需求文档

## 1. 背景
当前 VisionAgent 的报告能力主要依赖导出 HTML 和 JSON 文件。随着客户诉求升级，静态导出报告已经无法满足在线查看、连续复检、状态管理、筛选联动等需求。

为避免客户访问内部巡检生产系统，同时保持内部能力稳定，计划新增独立项目 `report-system`，专门面向客户提供在线报告访问能力。

## 2. 项目目标
1. 新建独立的在线报告系统，供客户通过浏览器访问。
2. 由 VisionAgent 负责“发布报告”，由 Report System 负责“接收、存储、展示报告”。
3. 不共享完整业务后台，不直接暴露 VisionAgent 内部巡检能力。
4. 将当前“运营执行版报告”逐步升级为正式在线产品能力。
5. 图片不再依赖本地文件，统一通过阿里云 OSS 提供访问。

## 3. 系统边界
### 3.1 VisionAgent 职责
1. 负责巡检生产。
2. 负责整理已冻结的报告快照。
3. 负责将报告数据与图片引用发布到 Report System。

### 3.2 Report System 职责
1. 接收 VisionAgent 发布的报告数据。
2. 自行存储报告主体、门店、图片、问题项等结构化数据。
3. 面向客户提供在线查询、查看、复检、状态管理能力。
4. 不承担抓拍、上传、巡检、技能配置、计划管理等后台职责。

## 4. 核心原则
1. 两套系统职责清晰分离。
2. 两者通过“发布后的报告数据”对接，而不是共享完整业务后台。
3. Report System 不直接依赖 VisionAgent 内部 SQLite 业务表结构。
4. 报告发布必须是“快照”，而不是实时回查生产数据。
5. 图片统一走 OSS，不再要求报告系统读取 VisionAgent 本地图片目录。

## 5. 报告发布模型
## 5.1 报告不是导出文件，而是发布对象
后续在 VisionAgent 中，原“导出报告”能力需要逐步演进为“发布报告”。

一份报告发布对象至少包含：
1. 报告头信息。
2. 统计周期。
3. 企业信息。
4. 报告版本信息。
5. 汇总指标。
6. 问题门店列表。
7. 图片与问题项明细。
8. 图片 OSS 地址与元数据。

## 5.2 报告快照原则
1. 一旦发布，报告数据按版本冻结。
2. 客户看到的是某次发布版本，而不是生产库实时结果。
3. 若内部数据修复后需要客户重新查看，应重新发布新版本，而不是覆盖历史版本。

## 6. 图片存储方案
## 6.1 目标
图片不再以 VisionAgent 本地文件作为客户访问依赖，统一使用阿里云 OSS。

## 6.2 要求
VisionAgent 在发布报告时需完成：
1. 报告相关图片上传到 OSS。
2. 为每张图片生成稳定引用信息。
3. 发布到 Report System 的数据中带上图片元数据。

## 6.3 建议元数据
每张图片建议至少包含：
1. `object_key`
2. `bucket`
3. `region`
4. `url`
5. `width`
6. `height`
7. `captured_at`
8. `store_id`
9. `store_name`
10. `enterprise_id`

## 7. Report System 数据存储策略
## 7.1 基本要求
Report System 自己存储报告数据，不依赖 VisionAgent 的内部数据库。

## 7.2 第一阶段建议
第一阶段可以先使用 SQLite 快速落地，但必须清楚其边界：
1. 适合单机、低并发、快速验证产品形态。
2. 如果客户访问量、复检并发、日志要求上升，需预留迁移到 PostgreSQL/MySQL 的能力。

## 7.3 长期建议
如果确认作为长期客户系统，建议最终迁移到 PostgreSQL 或 MySQL。

## 8. 数据模型建议
Report System 应围绕“已发布报告”设计，而不是围绕巡检执行过程设计。

建议至少包含：
1. `report`
2. `report_store`
3. `report_image`
4. `report_issue`
5. `report_review_log`
6. `report_user`

其中：
1. `report` 到 `report_issue` 属于发布快照主体。
2. `report_image.review_status` 与 `report_review_log` 属于在线业务状态。

## 9. 发布接口设计要求
## 9.1 总体要求
VisionAgent 通过 API 向 Report System 发布报告。

## 9.2 接口要求
1. 发布接口必须支持幂等。
2. 同一份报告重复发布时，不能生成脏重复数据。
3. 需支持版本化。
4. 需支持发布结果回执。

## 9.3 建议字段
发布请求建议至少包含：
1. `publish_id`
2. `source_system`
3. `source_enterprise_id`
4. `enterprise_name`
5. `report_type`
6. `report_version`
7. `start_date`
8. `end_date`
9. `operator`
10. `summary_metrics`
11. `stores`
12. `images`
13. `issues`
14. `published_at`

## 10. 客户端功能需求
## 10.1 登录与权限
1. 客户需登录后访问报告系统。
2. 每个账号只能访问授权范围内的企业、品牌、门店或报告。
3. 报告系统不提供巡检后台能力入口。

## 10.2 报告列表
1. 可查看已发布报告列表。
2. 支持按企业、日期、批次、状态筛选。
3. 支持进入报告详情页。

## 10.3 报告详情
1. 展示汇总指标。
2. 展示问题门店清单。
3. 展示问题图片。
4. 展示问题项与分析说明。
5. 支持按组织、门店、状态筛选。

## 10.4 连续复检
1. 用户可在图片详情中连续查看问题图片。
2. 支持“上一个”“下一个”。
3. 支持修改复检状态。
4. 需记录操作者与时间。

## 10.5 状态要求
建议至少支持：
1. 待复检
2. 已复检
3. 后续可扩展为：已确认、已忽略、待整改等

## 11. 非功能要求
1. 客户访问稳定。
2. 页面加载性能可接受。
3. 权限隔离清晰。
4. 报告数据可追溯。
5. 发布失败可重试。
6. 图片访问路径稳定。

## 12. 分批实施计划
### 批次一：项目初始化与数据契约定义
目标：明确系统边界，搭出可运行骨架。

内容：
1. 初始化 `report-system` 项目结构。
2. 定义发布接口协议。
3. 定义最小数据模型。
4. 搭建基础登录、报告列表、报告详情页面骨架。

验收：
1. 项目可启动。
2. 能接收一份 mock 报告数据。
3. 能查看基础报告列表与详情。

### 批次二：VisionAgent 发布报告能力接入
目标：实现从 VisionAgent 向 Report System 正式发布。

内容：
1. VisionAgent 增加“发布报告”动作。
2. VisionAgent 上传报告图片到 OSS。
3. VisionAgent 调用 Report System API 完成发布。
4. Report System 完成报告入库。

验收：
1. 一次发布可生成一份在线可查看报告。
2. 失败可重试。
3. 发布具备幂等性。

### 批次三：在线复检能力
目标：让客户真正开始用。

内容：
1. 门店问题清单。
2. 图片详情弹窗。
3. 上一个/下一个连续复检。
4. 复检状态存储。
5. 操作日志。

验收：
1. 客户可连续复检。
2. 复检状态可保存并再次读取。

### 批次四：正式运营准备
目标：提升可维护性与可交付性。

内容：
1. 补权限体系。
2. 补部署说明。
3. 评估 SQLite 是否需要迁移。
4. 完善审计与监控。

验收：
1. 系统可稳定部署。
2. 权限隔离生效。
3. 能支撑客户持续访问。

## 13. 第一阶段验收标准
1. Report System 可独立启动。
2. 能接收一份标准化报告发布数据。
3. 报告列表页和详情页可用。
4. 图片展示基于 OSS URL，而非本地文件目录。
5. 复检状态具备扩展存储位置。

## 14. 风险与注意事项
1. 不建议让 Report System 直接读取 VisionAgent 内部 SQLite 表。
2. 不建议继续以静态导出 HTML 作为在线系统主交付形式。
3. 图片访问策略必须尽早定清楚，尤其是 URL 是否长期有效。
4. 发布接口的幂等与版本策略必须在一期就定义清楚。
5. 如果后期客户访问量增长，SQLite 可能需要迁移。
