# Report System 单租户部署规范

## 1. 文档目的
本规范用于明确 `report-system` 的交付模型、部署边界、初始化范围和后续扩展原则。该系统不按强多租户模式建设，而是采用“单客户单实例”的部署策略。

## 2. 部署原则
1. 一个客户对应一个 `report-system` 实例。
2. 一个实例只服务一个客户，不混用其他客户数据。
3. 同一套代码基线可以复用，但配置、数据库、域名和品牌资源必须隔离。
4. 客户定制优先通过配置和扩展层实现，不直接污染主干业务逻辑。

## 3. 单租户实例的最小组成
每个客户实例至少应独立拥有：

1. 一套运行配置。
2. 一个独立数据库。
3. 一个独立域名或访问入口。
4. 一套客户品牌配置。
5. 一套日志和备份策略。

## 4. 配置范围
建议配置项至少包括：

1. `TENANT_ID`
2. `TENANT_NAME`
3. `BASE_URL`
4. `DATABASE_URL`
5. `BRAND_NAME`
6. `LOGO_URL`
7. `PRIMARY_COLOR`
8. `DEFAULT_TIMEZONE`
9. `ENABLE_FEATURES`
10. `SUPPORTED_PAYLOAD_VERSIONS`

## 5. 初始化内容
每次新客户部署时，应完成以下初始化：

1. 租户基础信息。
2. 品牌配置。
3. 默认角色和账号。
4. 报告展示基础配置。
5. 发布接口连通性验证。
6. 首批测试报告导入。

## 6. 数据隔离要求
1. 不允许多个客户共用同一套业务数据。
2. 不允许在一个实例中通过 `tenant_id` 硬隔离多个客户作为长期方案。
3. 如确需共用基础设施，也应保证数据库逻辑隔离和备份隔离。

## 7. 升级与回滚
1. 每个客户实例应支持独立升级。
2. 每个客户实例应支持独立回滚。
3. 升级前必须确认支持的 `payload_version`。
4. 如契约升级不兼容，应先完成试点客户灰度。

## 8. 定制边界

### 8.1 允许定制
1. Logo、主题色、品牌名称。
2. 页面文案。
3. 字段显隐和字段顺序。
4. 默认筛选项。
5. 客户专属展示模块开关。

### 8.2 不建议定制进主干
1. 单客户专属状态机。
2. 单客户专属主流程跳转。
3. 单客户专属报告计算口径。
4. 单客户专属数据库结构分叉。

## 9. 运维要求
1. 部署命令应标准化。
2. 日志目录、数据库路径、备份路径应可配置。
3. 必须保留实例级监控和错误告警能力。
4. 发布接收异常应可快速定位。

## 10. 验收标准
1. 新客户实例可以通过标准模板初始化。
2. 客户数据与其他实例完全隔离。
3. 首次发布报告可以成功接收并展示。
4. 客户品牌配置可以在不改主干代码的前提下生效。
5. 升级和回滚具备明确操作路径。
